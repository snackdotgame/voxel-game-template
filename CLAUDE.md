# Agent Guidelines for Snack.Game Projects

This is a Snack.Game project generated by the Snack CLI. Package scripts are the source of truth
for local development, checks, builds, and publishing.

## Project Shape

- `snack.json` configures game entrypoints, assets, and public game metadata.
- `src/client.ts` is browser code loaded by Vite through the Snack host shell.
- `src/server.ts` is server code loaded by the restricted Snack runtime.
- `src/shared/*` is for types, constants, and helpers shared by client and server code.
- `.snack/types/*` contains generated ambient declarations for `snack:client` and
  `snack:server`.

## Local Development

- Use the project's package manager scripts. Do not add Snack-owned install, typecheck, lint, or
  format commands.
- `dev` runs the Vite client dev server and `snack dev` server runtime together through Turbo.
- Always open and test the local game through the Snack host shell at
  `http://127.0.0.1:3030/`.
- Do NOT use the Vite-only client port `3031` as the game URL for networked play; it does not
  provide the required Snack launch envelope.
- Local ports can be configured in `snack.json` (`"dev": { "port": 3030, "clientPort": 3031 }`),
  via `snack dev --port` / `--client-port`, or with the `SNACK_DEV_PORT`, `SNACK_CLIENT_PORT`,
  and `SNACK_CLIENT_HOST` environment variables. Flags and environment take precedence over
  `snack.json`.
- Run `check` before committing gameplay changes and `build` before publishing.

## Runtime Contract

- Client code imports `client` from `snack:client`.
- Server code imports `server` from `snack:server` and exports `main()`.
- Server code can read `server.config` for the effective per-session JSON config snapshot.
- Use `recv()` and `drain()` on datagram or stream channels.
- Do not use callback-style receive handlers or raw WebTransport/postMessage plumbing.
- Keep authoritative multiplayer state in server code. Treat browser state as display/input state.
- Validate network messages before trusting them.
- The server runtime is restricted. Do not use filesystem, process, native addons, raw sockets,
  server listeners, or outbound `fetch` unless Snack explicitly supports that capability.

## Publishing

- `publish` is the creator-facing command for building, uploading, and activating a game version.
- `deploy` is lower-level and mainly useful for debugging or CI upload flows.
- When asked to publish, deploy, release, or prepare the game for hosting, use the generated
  `snack-game-publish` skill before running commands.
- Use `snack auth whoami` to check authentication and `snack auth login` when credentials are
  missing.
- Configure public game metadata in `snack.json`, including `game.maxPlayers`, optional
  `game.titleImage`, and optional `game.serverConfigSchema` for server config fields.
