# Agent Guidelines for Snack.Game Projects

This is a Snack.Game project generated by the Snack CLI. Package scripts are the source of truth
for local development, checks, builds, and upload-oriented publishing workflows. Direct
`snack publish` and `snack preview` commands are also valid when using version ids or flags.

## Project Shape

- `snack.json` configures game entrypoints, assets, and public game metadata.
- `src/client.ts` is browser code loaded by Vite through the Snack host shell.
- `src/server.ts` is server code loaded by the restricted Snack runtime.
- `src/shared/*` is for types, constants, and helpers shared by client and server code.
- `.snack/scaffold.json` records the generated project scaffold version.
- `.snack/types/*` contains generated ambient declarations for `snack:client` and
  `snack:server`.

## Local Development

- Use the project's package manager scripts. Do not add Snack-owned install, typecheck, lint, or
  format commands.
- `dev` runs the Vite client dev server and `snack dev` server runtime together through Turbo.
- Always open and test the local game through the Snack host shell at
  `http://127.0.0.1:3030/`.
- Do NOT use the Vite-only client port `3031` as the game URL for networked play; it does not
  provide the required Snack launch envelope.
- Local ports can be configured in `snack.json` (`"dev": { "port": 3030, "clientPort": 3031 }`),
  via `snack dev --port` / `--client-port`, or with the `SNACK_DEV_PORT`, `SNACK_CLIENT_PORT`,
  and `SNACK_CLIENT_HOST` environment variables. Flags and environment take precedence over
  `snack.json`.
- Run `check` before committing gameplay changes and `build` before publishing.
- After updating the project-local Snack CLI dependency, run `snack scaffold upgrade` from a clean
  Git worktree to refresh Snack-owned project infrastructure, generated types, and generated skills.

## Runtime Contract

- Client code imports `client` from `snack:client`.
- Server code imports `server` from `snack:server` and exports `main()`.
- Server code can read `server.config` for the effective per-session JSON config snapshot.
- Use `recv()` and `drain()` on datagram, stream, or chat channels.
- Route human-visible player chat through `client.chat` and `server.chat`, not gameplay datagrams or
  streams. Clients send only to the authoritative server; only server code selects recipients with
  `server.chat.send(..., { only, except })`.
- Do not use callback-style receive handlers or raw WebTransport/postMessage plumbing.
- Keep authoritative multiplayer state in server code. Treat browser state as display/input state.
- Validate network messages before trusting them.
- The server runtime is restricted. Do not use filesystem, process, native addons, raw sockets,
  server listeners, or outbound `fetch` unless Snack explicitly supports that capability.

## Resource Discipline

- Hosted game sessions have enforced CPU and server JavaScript heap limits.
- Keep loops, queues, histories, caches, timers, and per-player state explicitly bounded.
- Never let client-controlled message volume determine unbounded work or allocation.
- Avoid busy loops. Use `server.elapsedMs()`, `server.sleep()`, blocking channel receives, fixed
  deadlines, and bounded catch-up.
- Clean up state on disconnect and fresh-launch rejoin. Coalesce or drop superseded noncritical data
  instead of accumulating it.
- Test performance with realistic player counts. Use `snack-debug-performance` for current public
  limits, measurement, and optimization guidance.

## Bundled Skills

Load only the skill needed for the current task:

- `snack-configure-project` for `snack.json`, scripts, entrypoints, and server config.
- `snack-design-gameplay` for mechanics, rules, progression, encounters, and the playable loop.
- `snack-design-game-ux` for HUDs, menus, responsive layout, controls, and accessibility.
- `snack-build-text-chat` for human-visible global, team, proximity, lobby, spectator, or system
  chat using `client.chat` and `server.chat`.
- `snack-build-multiplayer` to select and implement turn-based, snapshot interpolation, client
  prediction, rollback, or lag compensation.
- `snack-3d-physics` for 3D physics, rigid bodies, collision, character movement, or any physical
  simulation via `jolt-ts`; load it alongside `snack-build-multiplayer` for deterministic networked
  simulation.
- `snack-design-binary-protocol` for binary wire formats, bitpacking, quantization, delta
  compression, priority accumulators, and the ~1,000-byte datagram budget; load it alongside the
  selected multiplayer approach.
- `snack-threejs-rendering` for Three.js rendering, assets, cameras, lighting, and cleanup.
- `snack-debug-performance` for measured client, server, render, memory, and network diagnosis.
- `snack-playtest-game` for host-shell multiplayer, network simulation, device, and completion
  checks.
- `snack-game-publish` for push, preview, and publish workflows.

## Publishing

- `push` builds, uploads, and finalizes a new immutable version without publishing or launching it.
- `versions` lists pushed builds with compatibility version numbers, version names, and version ids
  for the current game.
- `publish [version-id]` makes a pushed version public and live. When omitted, the version id stored
  in `game.versionId` is used.
- `preview [version-id]` launches an unlisted server. When omitted, the latest ready pushed version is
  used. Previewing requires `game.genre` in `snack.json`.
- When asked to publish, preview, release, or prepare the game for hosting, use the generated
  `snack-game-publish` skill before running commands.
- Use `snack auth whoami` to check authentication and `snack auth login` when credentials are
  missing.
- Configure public game metadata in `snack.json`, including `game.maxPlayers`, required
  `game.titleImage`, `game.versionId` written by `snack push`, optional `game.versionName` for the
  next pushed version, `game.genre` before previewing or launching hosted servers, optional
  `game.platforms` for supported device classes, and optional `game.serverConfigSchema` for server
  config fields.
